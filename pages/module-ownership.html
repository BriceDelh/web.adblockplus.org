title=Module ownership

{{s1 We're using a module ownership system similar to what Mozilla has. While most modules are areas of our code base, some have nothing to do with the code, but are governed in a very similar way.}}

{{s2 The module owner role}}

{{s3 Module owners have functional authority and responsibility over their area. Only the respective super reviewer can overrule their decisions, but this should not be necessary in practice.}}

{{s4 Responsibilities for code module owners}}

  • {{s5 Review and approve all changes to the module}}
  • {{s6 Grant or revoke commit access}}
  • {{s7 Maintain and improve quality}}
  • {{s8 Set the priority of new issues and reject them if necessary}}
  • {{s9 Ensure all high priority issues get tackled}}

{{s10 Responsibilities for non-code module owners}}

  • {{s11 Oversee all activities}}
  • {{s12 Plan future activities}}
  • {{s13 Ensure good communication with other modules as well as non-developers}}

{{s14 Super reviews}}

{{s15 Each module has a designated super reviewer, who will oversee things at a higher level.}}

{{s16 Super reviews for code modules}}

{{s17 While the module owner will review all changes, certain changes should also be reviewed by the super reviewer, in particular:}}

  • {{s18 API changes (both public API as well as API between modules)}}
  • {{s19 Major refactoring}}
  • {{s20 New subprojects}}

{{s21 For these types of changes, the module owner should review first and make sure that all of their comments are addressed, then request a super review.}}

{{s22 Super reviews for non-code modules}}

{{s23 There are no reviews for anything happening in non-code modules, but the super reviewers will oversee major projects and communication with other modules and non-developers in a similar fashion. The owner should report to the respective super reviewer on a regular basis.}}
