title=Writing Adblock Plus filters

{{s1 Current Adblock Plus versions allow you to "tweak" your filters in many different ways. This document explains the choices that you have and how they can be used.}}

{{s2 Disclaimer: All filter examples given here are really only examples and are not meant to be used.}}

{{s3 Introduction to Adblock Plus filters}}

{{s4 The options described in this section should be enough for users who have to create a filter occasionally.}}

{{s5 Basic filter rules}}

{{s6 The most trivial filter you can define is of course the address of banner you want to block. However, often this address changes every time you open a page. For example it could be}} http://example.com/ads/banner123.gif {{s7 where 123 is a random number. Here blocking the complete address won't help you, you need a more general filter — like}} http://example.com/ads/banner*.gif{{s8 . Or maybe even}} http://example.com/ads/*{{s9 .}}

{{s10 Note: Make sure that you are not replacing too much by wildcards. The filter}} http://example.com/* {{s11 will definitely block all banners but it will also block everything else from example.com that you still might want to see.}}

{{s12 Defining exception rules}}

{{s13 Sometimes you will notice that one of your filters that is usually working quite well blocks in some case blocks something that it shouldn't be blocking. You don't want to remove this filter but you still don't want it to match in this one case.}}

{{s14 That's what exception rules are good for — they allow you to define cases where filters shouldn't be applied. For example if you are unhappy with your filter}} adv {{s15 blocking}} http://example.com/advice.html{{s16 , you can define an exception rule}} @@advice{{s17 . Exception rules are no different from filter rules, you can use wildcards or regular expressions. You only have to precede them by}} @@ {{s18 to indicate an exception rule.}}

{{s19 Exception rules can do more. If an exception rule starts with http:// or https:// (optionally with a pipe before it) it will make whole pages an exception. For example, if your exception rule is}} @@|http://example.com {{s20 and you open some page from example.com — Adblock Plus will be entirely disabled on this page and nothing will be blocked.}}

{{s21 Matching at beginning/end of an address}}

{{s22 Usually Adblock Plus treats every filter as if it had a wildcard at its beginning and end, e.g. there is not difference between the filters}} ad {{s23 and}} *ad*{{s24 . While this is usually unproblematic, sometimes you wish that the filter you defined only matches at the beginning or end of an address. For example you might want to block all Flash, but if you add the filter}} swf {{s25 the address}} http://example.com/swf/index.html {{s26 will also be blocked.}}

{{s27 Solution to this problem: add a pipe symbol to the filter to show that there should be definitely the end of the address at this point. For example the filter}} swf| {{s28 will block}} http://example.com/annoyingflash.swf {{s29 but not}} http://example.com/swf/index.html{{s30 . And the filter |http://baddomain.example/ will block http://baddomain.example/banner.gif but not http://gooddomain.example/analyze?http://baddomain.example.}}

{{s31 Sometimes one wants to block}} http://example.com/banner.gif {{s32 as well as}} https://example.com/banner.gif {{s33 and}} http://www.example.com/banner.gif{{s34 . This can be achieved by putting two pipe symbols in front of the filter which makes sure the filter matches at the beginning of the domain name:}} ||example.com/banner.gif {{s35 will block all these addresses while not blocking http://badexample.com/banner.gif or http://gooddomain.example/analyze?http://example.com/banner.gif (requires Adblock Plus 1.1 or higher).}}

{{s36 Marking separator characters}}

{{s37 Often you need to accept any separator character in a filter. For example, you might write a filter that blocks}} http://example.com/ {{s38 and}} http://example.com:8000/ {{s39 but not}} http://example.com.ar/{{s40 . Here the symbol ^ can be used as a placeholder for a single separator character:}} http://example.com^ {{s41 (requires Adblock Plus 1.1 or higher).}}

{{s42 Separator character is anything but a letter, a digit, or one of the following: _ - . %. The end of the address is also accepted as separator. In the following example all separator characters are shown in red:}} http://example.com:8000/foo.bar?a=12&b=%D1%82%D0%B5%D1%81%D1%82{{s43 . So this address can be blocked with the filter}} ^example.com^ {{s44 or}} ^%D1%82%D0%B5%D1%81%D1%82^ {{s45 or}} ^foo.bar^{{s46 .}}

{{s47 Comments}}

{{s48 Any rule that starts with an exclamation mark is considered a comment. It will still show up in the filter list but in grey instead of black. Adblock Plus will ignore this rule for actual blocking so it is safe to write there whatever you want. You can place a comment rule above a real filter to describe what it is doing. Or you can put a comment on top of your filter list stating your authorship (most filter list authors do that).}}

{{s49 Special comments}}

{{s50 Special comments will only have an effect in downloaded filter lists, not in custom filters.}} {{s51 They can set a number of parameters for the filter list:}}

  • ! Homepage: http://example.com/

    {{s52 This comment determines which webpage should be linked as filter list homepage.}}

  • ! Title: FooList

    {{s53 This comment sets a fixed title for the filter list.}} {{s54 If this comment is present the user will no longer be able to change the title.}}

  • ! Expires: 5 days

    {{s55 This comment sets the update interval for the filter list, the value can be given in days (e.g.}} 5 days{{s56 ) or hours (e.g.}} 8 hours{{s57 ).}} {{s58 Any value between 1 hour and 14 days is possible.}} {{s59 Note that the update will not necessarily happen after this time interval.}} {{s60 The actual update time is slightly randomized and depends on some additional factors to reduce server load.}}

  • ! Checksum: OaopkIiiAl77sSHk/VAWDA

    {{s61 This comment makes sure that accidental corruption of the data won't result in broken filters.}} {{s62 For example, some firewall software might modify the filter */adnetwork/* on download in an attempt to protect the user against ads.}} {{s63 It will remove part of the filter so that Adblock Plus will only see the filter}} **{{s64 .}} {{s65 A checksum comment in the filter list protects against this scenario, any modifications will have the result that the checksum no longer matches and Adblock Plus will ignore the data.}}

    {{s66 To calculate the checksum the following steps need to be performed:}}

    • {{s67 Remove the existing checksum comment if any.}}
    • {{s68 Encode filter list text using UTF-8 encoding.}}
    • {{s69 Convert all line breaks to Unix style (replace}} \r {{s70 by}} \n {{s71 if present).}}
    • {{s72 Remove empty lines (replace sequences of the}} \n {{s73 character by a single}} \n {{s74 character).}}
    • {{s75 Calculate the base64-encoded MD5 checksum of the text, remove trailing}} = {{s76 characters if any.}}

    {{s77 You can also have a look at the Python-based reference implementations to validate a checksum and to add a checksum to a file.}}

  • ! Redirect: http://example.com/list.txt

    {{s78 This comment indicates that the filter list has moved to a new download address.}} {{s79 Adblock Plus will ignore any file contents beyond that comment and immediately try downloading from the new address.}} {{s80 In case of success the address of the filter list will be updated in the settings.}} {{s81 This comment is ignored if the new address is the same as the current address, meaning that it can be used to enforce the "canonical" address of the filter list.}}

  • ! Version: 1234

    {{s82 This comment defines a numerical version of the filter list.}} {{s83 This version number will be displayed in issue reports and can be used to verify that the report refers to the current version of the filter list.}}

{{s84 Advanced features}}

{{s85 The features described in this section are usually used only by power users and filterlist creators. Feel free to skip it.}}

{{s86 Specifying filter options}}

{{s87 Adblock Plus allows you to specify a number of options to modify the behavior of a filter. You list these options separated with commas after a dollar sign ($) at the end of the filter, for example:}}

*/ads/*$script,match-case

{{s88 Here}} */ads/* {{s89 is the actual filter and}} script {{s90 and}} match-case {{s91 are its options. Currently the following options are supported:}}

  • {{s92 Type options: determine which types of elements a filter can block (or whitelist in case of an exception rule). Multiple type options can be specified to indicate that the filter should be applied to several types of elements. Possible types are:}}
    • script{{s93 -- external scripts loaded via HTML script tag}}
    • image{{s94 -- regular images, typically loaded via HTML img tag}}
    • stylesheet{{s95 -- external CSS stylesheet files}}
    • object{{s96 -- content handled by browser plugins, e.g. Flash or Java}}
    • xmlhttprequest{{s98 -- requests started by the XMLHttpRequest object}}
    • object-subrequest{{s99 -- requests started plugins like Flash}}
    • subdocument{{s100 -- embedded pages, usually included via HTML frames}}
    • document{{s101 -- the page itself (only exception rules can be applied to the page)}}
    • elemhide{{s102 -- for exception rules only, similar to}} document {{s103 but only disables element hiding rules on the page rather than all filter rules (Adblock Plus 1.2 and higher required)}}
    • other{{s104 -- types of requests not covered in the list above}}
    {{s105 The type options}} background{{s106 ,}} xbl{{s107 ,}} ping {{s108 and}} dtd {{s109 are outdated and should no longer be used.}}
  • {{s110 Inverse type options: specify the element types the filter should not be applied to. Possible inverse type options:}} ~script{{s111 ,}} ~image{{s112 ,}} ~stylesheet{{s113 ,}} ~object{{s114 ,}} ~xmlhttprequest{{s115 ,}} ~object-subrequest{{s116 ,}} ~subdocument{{s117 ,}} ~document{{s118 ,}} ~elemhide{{s119 ,}} ~other
  • {{s120 Restriction to third-party/first-party requests: If the}} third-party {{s121 option is specified, the filter is only applied to requests from a different origin than the currently viewed page. Similarly,}} ~third-party {{s122 restricts the filter to requests from the same origin as the currently viewed page.}}
  • {{s123 Domain restrictions: The option}} domain=example.com {{s124 means that the filter should only be applied on pages from "example.com" domain. Multiple domains can be specified using "|" as separator: with the option}} domain=example.com|example.net {{s125 the filter will only be applied on pages from "example.com" or "example.net" domains. If a domain name is preceded with "~", the filter should not be applied on pages from this domain. For example,}} domain=~example.com {{s126 means that the filter should be applied on pages from any domain but "example.com" and}} domain=example.com|~foo.example.com {{s127 restricts the filter to the "example.com" domain with the exception of "foo.example.com" subdomain.}}
  • {{s128 Sitekey restrictions: The option}} sitekey=abcdsitekeydcba {{s129 means that the filter should only be applied on pages that provide a public key and a signature which can be verified by that very same public key that is also contained in the filter (but without the trailing =). Multiple sitekeys can be specified using "|" as separator: with the option}} sitekey=abcdsitekeydcba|bcdesitekeyedcb {{s130 the filter will only be applied on pages providing either sitekey "abcdsitekeydcba" or "bcdesitekeyedcb". This is similar to domain restrictions but allows covering scenarios where a single filter should apply to a very large number of domains. Note that sitekey restrictions require modifications on the server-side.}}
  • match-case{{s131 -- makes the filter only apply to addresses with matching letter case, e.g. the filter}} */BannerAd.gif$match-case {{s132 will block}} http://example.com/BannerAd.gif {{s133 but not}} http://example.com/bannerad.gif{{s134 .}}
  • collapse{{s135 -- this option will override the global "Hide placeholders of blocked elements" option and make sure the filter always hides the element. Similarly the}} ~collapse {{s136 option will make sure the filter never hides the element.}}
  • donottrack{{s137 -- for any address matching a blocking rule with this option and not matching any exception rules with this option a Do-Not-Track header will be sent (requires Adblock Plus 1.3.5 or higher).}} {{s138 For backwards compatibility it is recommended to use this option in combination with contradicting type options, this will prevent this filter from blocking anything in earlier Adblock Plus versions:}} *$donottrack,image,~image

{{s139 Using regular expressions}}

{{s141 If you want even more control about what your filters match and what they don't match, you can use regular expressions. For example the filter}} /banner\d+/ {{s142 will match}} banner123 {{s143 and}} banner321 {{s144 but not}} banners{{s145 . You can check out documentation on regular expressions to learn how to write them.}}

{{s146 Note: For performance reasons it is recommended not to use regular expressions if they can be avoided.}}

{{s147 Element hiding}}

{{s148 Basic rules}}

{{s149 Sometimes you will find advertisements that can't be blocked because they are embedded as text in the web page itself. If you look at the source code of the web page you might find something like this:}}

<div class="textad">
Cheapest tofu, only here and now!
</div>
<div id="sponsorad">
Really cheap tofu, click here!
</div>
<textad>
Only here you get the best tofu!
</textad>

{{s150 You need to download the web page so you will necessarily download the advertisements. All you can do here is to hide the advertisement so you don't need to see it. That's what element hiding is meant for.}}

{{s151 The first advertisement above is contained inside a div element with class attribute "textad". The following rule will hide exactly this combination:}} ##div.textad{{s152 . Here ## marks an element hiding rule while the rest is a selector identifying the elements that need to be hidden. You can hide elements by their id attribute similarly,}} ##div#sponsorad {{s153 will hide the second advertisement. You don't need to specify the element name, the rule}} ##*#sponsorad {{s154 will work just as well. And you can hide elements by element name only, e.g.}} ##textad {{s155 for the third advertisement.}}

{{s156 The Element Hiding Helper extension helps selecting the correct element and writing the corresponding rule without having to view the source code of the page. Basic HTML knowledge is useful nevertheless.}}

{{s157 Note: Element hiding works very differently from normal filters. This has the implication that no wildcards are supported in element hiding rules.}}

{{s158 Limiting rules to certain domains}}

{{s159 Usually you want to hide a specific ad on one specific site, you don't want your rule to be applied on other sites. For example the rule}} ##*.sponsor {{s160 might hide valid code on some sites. But if you write it as}} example.com##*.sponsor {{s161 it will be applied on}} http://example.com/ {{s162 and}} http://something.example.com/ {{s163 but not on}} http://example.org/{{s164 . You can also specify multiple domains — simply separate them with commas:}} domain1.example,domain2.example,domain3.example##*.sponsor{{s165 .}}

{{s166 If a domain name is preceded with "~", the rule will not be applied on pages from this domain (requires Adblock Plus 1.1 or higher). For example,}} ~example.com##*.sponsor {{s167 will be be applied on pages from any domain but "example.com" and}} example.com,~foo.example.com##*.sponsor {{s168 makes the rule apply on "example.com" domain with the exception of "foo.example.com" subdomain.}}

{{s169 Note: Due to the way how element hiding is implemented, you really can only limit it to full domain names. You cannot use any other part of the address and you cannot use}} domain {{s170 as a replacement for}} domain.example,domain.test{{s171 .}}

{{s172 Note: Element hiding rules with domain limitation can be used to hide browser's user interface elements as well. For example the filter rule}} browser##menuitem#javascriptConsole {{s173 will hide the JavaScript Console entry in Firefox's Tools menu.}}

{{s174 Attribute selectors}}

{{s175 Some advertisers don't make it easy for you — their text advertisements have neither an id nor a class attribute. You can use other attributes to hide those, for example}} ##table[width="80%"] {{s176 will hide tables with width attribute set to 80%. If you don't want to specify the full value of the attribute,}} ##div[title*="adv"] {{s177 will hide all div elements with title attribute containing the string "adv". You can also check the beginning and the end of an attribute, for example}} ##div[title^="adv"][title$="ert"] {{s178 will hide div elements with title starting with "adv" and ending with "ert". As you see, you can also use multiple conditions —}} table[width="80%"][bgcolor="white"] {{s179 will match tables with width attribute set to 80% and bgcolor attribute set to white.}}

{{s180 Advanced selectors}}

{{s182 In general, any CSS selector supported by Firefox can be used for element hiding. For example the following rule will hide anything following a div element with class "adheader":}} ##div.adheader + *{{s183 . For a full list of CSS list see W3C CSS specification (note that not all selectors are supported by Firefox yet).}}

{{s184 Note: This functionality is for advanced users only, you should be comfortable with CSS selectors to use it. Adblock Plus won't be able to check the syntax of the selector you are adding, if you use invalid CSS syntax you might break other (valid) rules you have. Check JavaScript Console for CSS errors.}}

{{s185 Exception rules}}

{{s186 Exception rules can disable existing rules on particular domains.}} {{s187 These are mostly useful to filter subscription authors who are extending another filter subscription that they cannot change.}} {{s188 For example, the rule}} ##div.textad {{s189 can be disabled on}} example.com {{s190 using the exception rule}} example.com#@#div.textad{{s191 .}} {{s192 The combination of these two rules has exactly the same effect as the single rule}} ~example.com##div.textad{{s193 .}} {{s194 It is recommended that you use exception rules only when you cannot change an overly general element hiding rule, in all the other cases limiting this rule to the necessary domains is preferable.}}

{{s195 Simplified element hiding syntax}}

{{s196 Adblock Plus supports simplified element hiding syntax (e.g.}} #div(id=foo){{s197 ) for backwards compatibility only. Using this syntax is discouraged, usual CSS selectors are preferred. Support for this syntax might be removed at some point.}}

{{s198 Implementing a sitekey on the server}}

{{s199 For a sitekey-restricted filter to apply, a webpage needs to return base64-encoded versions of the public key and a signature which Adblock Plus can validate. Currently, this means including them in both the HTTP response header (}}X-Adblock-Key: abcdpublickeydcba_abcdsignaturedcba{{s200 ) and the root tag of the document (}}<html data-adblockkey="abcdpublickeydcba_abcdsignaturedcba">{{s201 ).}}

{{s202 First you need to create a private RSA key (preferably 512 bit to keep the transfer volume low) and then a DER representation of the public key.}}

{{s203 The data used for creating the signature is a concatenated list of request variables (namely URI, host and user agent) separated by the}} NUL {{s204 character "\0". For example:}} 

  /index.html?q=foo\0www.example.com\0Mozilla/5.0 (X11; Ubuntu; Linux x86_64; rv:30.0) Gecko/20100101 Firefox/30.0

{{s205 Finally, generate the signature for this string by using the signature algorithm SEC_OID_ISO_SHA_WITH_RSA_SIGNATURE (default when using OpenSSL).}}
